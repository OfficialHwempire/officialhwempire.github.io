# Rest가 등장한 이유

## 1. REST 이전 웹 API SOAP
SOAP(Simple Object Access Protocol) 는 REST 이전에 존재했던 표준화된 프로토콜입니다. 

- XML 기반의 메세지 포맷
- 보안, 트랜잭션, 인증 등의 엔터프라이즈 급 기능 제공
- 표준 규격이 굉장히 많고 엄격
- 무겁고 규모가 큰 프로젝트에 적합
- 상당히 어려운 개발 난이도 (높은 개발 비용 )

SOAP는 기업 전용 대규모 프로젝트에 적합했습니다. 보안, 트랜잭션 처리를 대신 해주고, XML 기반으로 포맷이 규격화
되어 있어 모든 언어에서 동일하게 프로토콜을 적용가능했습니다. 하지만 개발 비용 및 처리 속도가 굉장히 무거워 소규모
프로젝트에는 용이하지 않은 단점이 있었습니다.

## 2. REST API 의 차별점

- Http 특화 아키텍처 (url, http method, 상태 코드 , 캐싱)
- JSON 기반 데이터 포맷 (XML 보다 빠름)
- 단순하고 직관적인 구조 (개발 비용이 낮음)
- 브라우저 및 모바일의 가벼운 환경에 최적화

REST api 와 기존 SOAP api의 차이점은 경량화 및 단순함입니다. REST API는 json 기반으로 작동해 가독성 및 데이터 처리
속도가 월등히 SOAP 보다 빠릅니다. 또한 Http 메서드를 그대로 활요해 soap 표준 규격 보다 단순합니다. 브라우저가 메인인
웹 환경에서 rest의 가벼운 특징과 쉽게 구현 가능한 점은 커다란 장점이 었습니다. 


## 3. REST API 의 장점

- 다양한 포맷및 파일 확장자 처리 가능
- 로컬 캐싱 가능
- 개발 난이도가 낮음

## 4. REST API 의 단점

- 다양한 설계 개방성으로 인한 일관성 결여
- Stateless 구조로 인한 복잡한 트랜잭선 처리 난해함

## 5. REST API 에 대한 개인적인 생각

REST API는 경량화 및 단순화라는 시대의 흐름에 따라 전환되었다고 생각합니다. 확장 가능성이 높고
구현 난이도가 낮은 REST API는 웹 브라우저 환경에서 매우 강력한 장점을 가지고 있습니다.
